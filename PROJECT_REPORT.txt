1. PROJECT OVERVIEW
-------------------
ArbPro is an AI-powered cryptocurrency arbitrage bot that detects and exploits
price discrepancies between Binance and Bybit exchanges on the BTC/USDT pair.
The system runs entirely on localhost and consists of a Python FastAPI backend
with an AI decision engine, and a React/TypeScript web dashboard.

================================================================================

2. ARCHITECTURE OVERVIEW
------------------------

  [A] BACKEND (Python / FastAPI)
      - api.py              : SOLE active REST + WebSocket API (port 8000)
      - main.py             : Entry point; starts API thread, keeps alive
      - predictor.py        : GRU neural network inference (PyTorch)
      - train_gru.py        : Offline model training script
      - core/server.py      : Legacy bot loop (no longer launched by main.py)
      - core/database.py    : SQLite trade persistence layer
      - core/risk_engine.py : Risk management logic
      - execution/trader.py : CCXT trade executor (Binance + Bybit testnet)
      - llm/ai_agent.py     : LLM decision engine (Gemini + Groq fallback)
      - llm/chatbot.py      : Secondary LLM chatbot interface

  [B] FRONTEND (React 18 / TypeScript / Vite)
      - frontend/           : Standardized to lowercase (single directory)
      - pages/Login.tsx              : Auth screen
      - pages/Index.tsx              : Main dashboard (WebSocket consumer)
      - components/CandleChart.tsx   : Live candlestick chart
      - components/SpreadChart.tsx   : Spread visualization
      - components/AIAssistant.tsx   : LLM message feed
      - components/MetricCard.tsx    : KPI tiles
      - components/Settings.tsx      : Threshold slider + bot toggle
      - components/TradeApprovalModal.tsx : Manual trade approval (LIVE)

  [C] DATA & MODELS
      - arbitrage.db        : Trade history (SQLite)
      - operator_vault.db   : User accounts (SQLite)
      - gru_model.pth       : Trained GRU model weights (~97KB)
      - scaler_params.npy   : Min-max scaler for GRU input
      - model/data/         : 20+ historical coin CSV datasets

3. TECHNOLOGY STACK
-------------------
  Backend Framework   : FastAPI + Uvicorn
  Async Runtime       : asyncio (Python)
  Exchange Library    : CCXT (async_support)
  AI Predictor        : PyTorch (GRU - 2 layers, hidden_dim=50)
  LLM Engine          : Google Gemini 2.0 Flash (primary)
                        Groq / LLaMA 3.1 8B Instant (fallback)
  Auth / Security     : bcrypt (passlib) + JWT (python-jose) + .env secrets
  Database            : SQLite (2 separate databases)
  Frontend Framework  : React 18 + TypeScript + Vite
  UI Components       : Radix UI + TailwindCSS + shadcn/ui
  Charts              : Recharts
  Routing             : React Router v6

         -> Now calls: bybit.create_market_sell_order(symbol, amount)
  [DONE] Bybit HTTP proxy support added to CCXT instantiation in trader.py

  PHASE 3: UI/UX Wiring
  ----------------------
  [DONE] POST /api/trade/approve endpoint added (JWT-protected)
         -> Accepts { "decision": "APPROVE" | "REJECT" }
         -> Signals waiting WebSocket loop via asyncio.Event
  [DONE] Human-in-the-loop execution flow implemented:
         1. AI detects opportunity and recommends EXECUTE
         2. Backend broadcasts { type: "pending_trade", trade: {...} }
         3. Bot PAUSES and awaits operator decision (30s timeout)
         4. On APPROVE: real trade executes + logged to DB
         5. On REJECT or TIMEOUT: trade cancelled, AI chat notified
  [DONE] TradeApprovalModal fully wired in React dashboard (Index.tsx)
         -> Auto-opens on pending_trade WebSocket event
         -> "Approve Trade" button calls POST /api/trade/approve APPROVE
         -> "Reject" button calls POST /api/trade/approve REJECT
  [DONE] Bybit Oracle display bug fixed (was reading marketData.kraken,
         now correctly reads marketData.bybit)

7. KNOWN REMAINING LIMITATIONS
-------------------------------
  1. BYBIT CONNECTIVITY: Bybit calls still depend on an HTTP proxy being
     active at 127.0.0.1:1080. Without it, Bybit balance/trade will fail
     gracefully but won't execute. Configure .env with a working proxy.

  2. BYBIT PRICE: The WebSocket still simulates Bybit price via a fixed
     fluctuation formula (not fetching live Bybit ticker). Actual spread
     requires a real Bybit price feed to be functional.

  3. CORE/SERVER.PY: Still exists as a legacy file. It is no longer
     launched by main.py but has not been deleted. Can be archived.
